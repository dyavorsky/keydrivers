Getting Started with keydrivers

Overview

kda() runs up to eight variable importance methods in a single call and returns a combined importance matrix alongside per-method metadata. Variable types are detected automatically from R class, and each method selects the appropriate model for the outcome type.

This vignette uses the mtcars dataset (included in base R) to demonstrate three scenarios:

  1. Continuous outcome — all eight methods on mpg
  2. Binary outcome — selected methods on a binarized version of mpg
  3. Ordinal outcome — selected methods on a discretized version of mpg

Setup

library(keydrivers)
data(mtcars)

mtcars contains performance and design data for 32 cars. We use fuel efficiency (mpg) as the outcome and four mechanical predictors as key drivers: displacement (disp), horsepower (hp), weight (wt), and rear axle ratio (drat).

Formula Syntax

The basic formula is y ~ x1 + x2 + x3, where y is the outcome and x1, x2, x3 are the key drivers whose importance will be measured.

An optional pipe separator | partitions key drivers from control variables:

kda(y ~ x1 + x2 + x3 | z1 + z2, data = df, ...)

Control variables are included in every regression-based model but are not assigned importance scores — their contribution to model fit is held constant so that driver importance is assessed net of those controls. Bivariate correlations (corr) never include controls. Without controls, use standard formula syntax:

kda(y ~ x1 + x2 + x3, data = df, ...)

1. Continuous Outcome

With a continuous outcome, all eight methods are available. Regression-based methods fit a linear model (lm) and apportion R². By default (normalize = TRUE) each method’s scores are rescaled to sum to 100, so a score of 32 for wt under Shapley means weight accounts for 32% of the full model’s R². This common scale makes cross-method comparison meaningful; pass normalize = FALSE to recover raw scores.

Each method accepts a corresponding _params list for tuning. For example, rf_params = list(ntree = 1000) increases the number of trees for more stable importance estimates.

result <- kda(mpg ~ disp + hp + wt + drat,
              data       = mtcars,
              corr       = TRUE,
              beta       = TRUE,
              useful     = TRUE,
              jrw        = TRUE,
              shapley    = TRUE,
              shapex     = TRUE,
              randforest = TRUE,
              rf_params  = list(ntree = 1000),
              xgboost_   = TRUE,
              verbose    = FALSE)

2. Binary Outcome

We binarize mpg at its median to create a 0/1 outcome. Binary outcomes use logistic regression (glm with family = binomial) and McFadden pseudo-R² in place of linear regression and R². kda() detects the binary type automatically — no arguments change.

Note that Johnson Relative Weights (jrw) is available for binary outcomes but XGBoost uses binary:logistic rather than the regression objective.

mtcars$high_mpg <- as.integer(mtcars$mpg > median(mtcars$mpg))

result_bin <- kda(high_mpg ~ disp + hp + wt + drat,
                  data    = mtcars,
                  corr    = TRUE,
                  beta    = TRUE,
                  useful  = TRUE,
                  shapley = TRUE,
                  verbose = FALSE)

print(result_bin)

3. Ordinal Outcome

We discretize mpg into three ordered tiers. Ordinal outcomes use proportional odds regression (polr from MASS) and McFadden pseudo-R². The outcome must be an ordered factor — kda() will error on an unordered factor and stop with a clear message.

Ordinal predictor variables, if any, are automatically converted to integer ranks before being passed to any regression-based method. Nominal (unordered factor) predictors are blocked entirely.

mtcars$mpg_tier <- cut(mtcars$mpg,
                       breaks         = quantile(mtcars$mpg, c(0, 1/3, 2/3, 1)),
                       labels         = c("low", "mid", "high"),
                       include.lowest = TRUE)
mtcars$mpg_tier <- factor(mtcars$mpg_tier, ordered = TRUE)

result_ord <- kda(mpg_tier ~ disp + hp + wt + drat,
                  data    = mtcars,
                  corr    = TRUE,
                  beta    = TRUE,
                  useful  = TRUE,
                  shapley = TRUE,
                  verbose = FALSE)

4. Bootstrap Confidence Intervals

kda_boot() wraps kda() and runs it on B bootstrap resamples to produce percentile confidence intervals for every importance score. All method flags and _params arguments are passed through unchanged; verbose is suppressed automatically during resampling.

The seed argument makes the full run reproducible — it is set before both the full-data call and all bootstrap iterations, so stochastic methods (RF, XGBoost, SHAP) produce the same results each time. For stochastic methods, reduce per-iteration cost via smaller ntree, nrounds, or nsim when B is large.

result_boot <- kda_boot(mpg ~ disp + hp + wt + drat,
                        data       = mtcars,
                        B          = 200,
                        conf_level = 0.95,
                        seed       = 42,
                        corr       = TRUE,
                        beta       = TRUE,
                        randforest = TRUE,
                        rf_params  = list(ntree = 500))

print(result_boot)

5. Plotting

plot() dispatches on both kda and kda_boot objects and produces a horizontal dot plot via ggplot2. Drivers are ordered from most to least important on the vertical axis; each method gets its own color. For kda_boot objects, CI lines are drawn automatically behind each point.

The colors argument accepts a named or unnamed character vector to override the default palette; bg_color sets the panel and plot background.

# Point estimates from the continuous example
plot(result)
# Bootstrap result with CIs — custom colors and background
plot(result_boot,
     colors   = c(corr = "steelblue", beta = "tomato", rf = "forestgreen"),
     bg_color = "#f8f8f8")

The returned object is a ggplot and can be extended with any ggplot2 layer:

library(ggplot2)
plot(result) + ggtitle("Key Drivers of Fuel Efficiency")

6. Interpreting Results When Methods Disagree

Running eight methods and getting eight identical rankings would be suspicious. Some disagreement is normal and informative. The key is knowing what each method is measuring and what divergence between them is telling you.

Why methods differ

Each method answers a subtly different question:

Method What it measures
Correlations Raw bivariate association — each predictor vs. outcome in isolation
Betas Partial association — each predictor controlling for all others simultaneously
Usefulness Marginal R² contribution — fit loss when this variable is removed
JRW / Shapley Fair partition of total R² across predictors, averaging over all orderings
RF / XGBoost Non-linear and interaction-aware importance

These are genuinely different questions. A variable can rank high on one and low on another without either answer being wrong.

What divergence usually means

Correlations high, Shapley/usefulness low: the variable shares its predictive power with another predictor. Both are measuring the same underlying construct. JRW and Shapley will split the shared credit between them rather than double-counting.

Betas in a different order than correlations: a suppressor or multicollinearity pattern. A variable can have a modest bivariate correlation but become important once others are held constant (or vice versa). This is a real finding about the structure of the data, not a modeling artifact.

Tree methods (RF, XGBoost) diverge from linear methods (Shapley, betas, usefulness): the relationship is probably non-linear or involves interactions between predictors. The linear methods fit a straight line and apportion its R²; the tree methods are picking up curvature or conditional structure that the line misses.

Methods largely agree: convergent evidence. Variables that rank consistently high across multiple methods are the most defensible findings.

Investigating a linear/tree divergence

When Shapley and RF give meaningfully different rankings for a variable, work through these steps:

1. Plot each predictor against the outcome with a smooth. A LOESS curve that departs from a straight line flags non-linearity directly.

library(ggplot2)

for (v in c("disp", "hp", "wt", "drat")) {
  p <- ggplot(mtcars, aes(x = .data[[v]], y = mpg)) +
    geom_point() +
    geom_smooth(method = "loess", se = TRUE) +
    geom_smooth(method = "lm", se = FALSE, color = "red", linetype = "dashed") +
    ggtitle(v)
  print(p)
}

2. Partial dependence plots from the RF model. A PDP shows the RF’s view of the functional form for each predictor, marginalizing over all others. A curved or step-shaped PDP confirms non-linearity. A flat PDP for a predictor the RF ranks highly suggests an interaction (the variable matters, but only conditionally).

library(randomForest)
library(pdp)

rf_model <- randomForest(mpg ~ disp + hp + wt + drat, data = mtcars, ntree = 1000)
partial(rf_model, pred.var = "wt", plot = TRUE, rug = TRUE)

3. Compare linear model R² to a GAM. A GAM fits a smooth function to each predictor instead of a straight line. If the GAM’s R² is meaningfully higher, the linear model is missing real structure and the Shapley values based on it are understating variables with curved relationships.

library(mgcv)

m_lm  <- lm(mpg ~ disp + hp + wt + drat, data = mtcars)
m_gam <- gam(mpg ~ s(disp) + s(hp) + s(wt) + s(drat), data = mtcars)

summary(m_lm)$r.squared
summary(m_gam)$r.sq   # materially higher = non-linearity matters

4. Test for interactions. If the PDPs look roughly linear but RF still diverges, the likely culprit is an interaction — one variable’s effect depends on the level of another. A 2D partial dependence plot reveals this: a non-flat surface indicates the two variables interact.

partial(rf_model, pred.var = c("wt", "hp"), plot = TRUE, chull = TRUE)

# Confirm with a formal test in the linear model
m_int <- lm(mpg ~ disp + hp + wt + drat + wt:hp, data = mtcars)
anova(m_lm, m_int)
